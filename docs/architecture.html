Overview

The project is an Arduino sketch that runs Conway's Game of Life on a Peggy 2.0 board — a 25×25 monochrome, multiplexed LED matrix. On power-up it seeds the RNG from analog-pin noise, fills a random first frame, then advances Conway's rules each step. When it decides the run is over — the field looped back to an earlier state, drifted past a generation cap, or settled into an unchanging population — it briefly shows the step count, reseeds, and repeats, so it runs autonomously forever.

The defining architectural property is that the simulation engine is fully decoupled from the display hardware behind a zero-cost grid abstraction. The same engine code compiles and runs both on the AVR (against a real Peggy 2 framebuffer) and on a development machine (against an in-memory bitmap), which is what makes the logic testable off-device.

Components

ConwayEngine core

Files: ConwayEngine.h / .cpp. The simulation itself, with no hardware knowledge.

Naming. Renamed from Peggy2ConwayEngine — despite the old Peggy2 prefix, the class has no hardware dependency (it talks only to ConwayGrid), so the misleading prefix was dropped (issue #29).
  • State two double-buffered ConwayGrid frames (currentGenIndex / nextGenIndex), plus a ring of compact 32-bit generation hashes (loopHashes, capacity loopHistorySize — 32 in the sketch, 128 in the emulator) and small scalar counters for the watchdog and stagnation backstops.
  • Config constructed from an EngineConfig aggregate (loopHistorySize, maxGenerations, populationWindow; any field 0 disables that heuristic). A convenience one-argument constructor sets only loopHistorySize, with both backstops off.
  • ComputeNextGen() records the current generation's hash (hashFrame, FNV-1a over the cells) — and its live-cell population when stagnation is enabled — then writes the next generation into the next frame; reads the current frame through the grid interface.
  • CommitNextGen() swaps the two frame indices (modulo frameCount = 2) and increments the committed-generation counter.
  • DetectLoop() hashes the just-computed frame and reports a loop if it matches any recent generation's hash (no full-frame comparison).
  • WatchdogExpired() true once the run has reached maxGenerations committed generations — the backstop for patterns that never recur within the hash window (see decisions).
  • PopulationStagnant() true once the live-cell count has been unchanged for populationWindow consecutive generations, tracked in O(1) with a run-length counter (no window buffer).
  • Rule (isAlive): survives on 2 live neighbours, born on 3, dead otherwise.
  • Neighbours (getNeighborCount/getCurrentCell): 8-neighbourhood with toroidal wrap (see decisions).

ConwayGrid abstraction seam

ConwayGrid.h is a compile-time typedef — no virtual dispatch, no per-cell cost — that selects a backend by build flag. GridDimensions.h defines ROWS/COLUMNS (25×25). The interface is just what the engine needs: HardwareInit, Clear, WritePoint, GetPoint, RefreshAll, Equals.

  • Peggy2Grid (Peggy2Grid.h) — on-device. A fully-inlined wrapper over a real Peggy2 framebuffer; forwards every call so it costs the same as direct use. Exposes a Device() escape hatch returning the underlying Peggy2* for display-only helpers.
  • HostGrid (host/HostGrid.h) — host builds (selected by -DCONWAY_HOST_BUILD). A plain in-memory bitmap (one unsigned long per row). Out-of-range access is dead/no-op; RefreshAll is a no-op.

Peggy2GameOfLife.ino orchestrator

The Arduino entry point. A SimpleTimer fires GameOfLifeStep() every STEP_PERIOD_MS (500 ms); loop() runs the timer and continuously calls RefreshAll on the current frame. SeedRandom() seeds the RNG via srand(analogRead(A0)). Display helpers (Line, PeggyWriter) are reached through GetCurrentFrame()->Device(). Holds the step/display state machine via the displayStepCounter flag, and ends a run when any of DetectLoop(), WatchdogExpired(), or PopulationStagnant() fire (engine configured with LOOP_HISTORY/MAX_GENERATIONS/POPULATION_WINDOW).

StepCounter support

Files: StepCounter.h / .cpp. Tracks the current run's step count and the all-time maximum since power-on. counterToString fills caller-owned fixed buffers — no per-call heap allocation, so an unattended run never leaks the AVR's small SRAM.

Runtime flow

The whole program is a small state machine driven by the timer. Compute is double-buffered: the next generation is written into nextGen while currentGen stays on the display, so the shown image never tears mid-step.

Why the engine refreshes during compute. The timer callback runs synchronously inside loop(), so loop()'s own refresh is paused for the entire ComputeNextGen pass. Because the Peggy 2 is a persistence-of-vision matrix, the engine refreshes the displayed frame once per row while computing the next one; otherwise the LEDs visibly dim/flicker each step. On HostGrid this refresh is a no-op.

Build & test paths

There are two independent build paths — that separation is the point of the grid abstraction.

Firmware (on-device)

Built and uploaded with the Arduino IDE from Peggy2GameOfLife.ino. Requires the Peggy2Serial (provides Peggy2, PeggyWriter) and SimpleTimer libraries in the Arduino libraries/ folder. The RefreshAll / multiplex display behaviour can only be validated here, on the board.

Host tests (off-device)

Pure host C++; needs only clang/gcc and make — no Arduino toolchain, no third-party libraries. From the repo root:

Key conventions & decisions

Zero-cost grid abstraction over the hardware

Rather than have the engine reach into a Peggy2 framebuffer directly, it talks to a ConwayGrid typedef resolved at compile time. This makes the engine testable on a host while keeping on-device code identical to direct calls (the README's long-standing wish to abstract the dependency "without a performance hit"). Trade-off: a build flag (CONWAY_HOST_BUILD) now selects the backend, and display-only code uses the Device() escape hatch.

Toroidal canvas via explicit modulo

Conway's rules assume an infinite plane; on the fixed 25×25 matrix the edges wrap (a torus), so e.g. (0,0) neighbours (24,24). The wrap uses signed coordinates and explicit (i + N) % N over ROWS/COLUMNS, so it behaves identically on the 16-bit-int AVR and a 32-bit-int host. (The original x == -1 test only wrapped by accident of 16-bit integer promotion and broke under host tests.) Making this configurable — toroidal vs. dead-border — is open work.

Termination: three independent heuristics, OR'd together

A run ends — and the field reseeds — as soon as any of three checks fire. Each is a separate EngineConfig tunable, and setting that tunable to 0 disables it.

  • Exact recurrence hashes each new generation (FNV-1a over the cells) and compares it against a ring of recent generation hashes. Storing a 32-bit hash instead of a whole frame makes a deep window cheap, so the sketch remembers 32 generations and the emulator 128 — enough to catch long-period recurrences the original 4-frame ring missed, such as a glider's 100-generation lap around the torus. Trade-off: a ~2-32 chance a hash collision ends a run one generation early, a harmless slightly-early reseed.
  • Max-generation watchdog (#8) reseeds after a fixed number of generations (1000 in the firmware) regardless of state, covering gliders/spaceships and chaotic or very-long-period fields that never recur within the hash window.
  • Population stagnation (#9) reseeds once the live-cell count has been unchanged for a long stretch (150 generations in the firmware), catching settled or stuck fields. Tracked in O(1) with a run-length counter rather than a per-window buffer, and kept well above the hash-history depth so ordinary oscillators are caught by exact recurrence first.

RNG seeded from analog noise, before the first frame

InitializeRandom() draws from stdlib rand(), so the same generator is seeded via srand(analogRead(A0)) — and seeded before the first frame is drawn — so each power-on differs. (Previously the only seeding call seeded Arduino's separate random(), leaving rand() unseeded and every power-on identical.)
